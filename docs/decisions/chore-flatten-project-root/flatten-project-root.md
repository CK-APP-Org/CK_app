# Chore: flatten the project root into the repo root

**Branch:** `chore/flatten-project-root` (requested ahead of Phase 4, outside the 8-phase refactor plan)

## What changed, in plain language

Before this change, the repo had two "roots" stacked on top of each other:
- **The repo root** (`D:\dev\CK_app\`) — where `.git`, `README.md`, `docs/`, `.github/` lived.
- **The actual Quasar app root** (`D:\dev\CK_app\CK_app\`) — a folder *named the same as the repo*, one level inside it, where `package.json`, `src/`, `quasar.config.js`, and everything else that makes the app run actually lived.

This meant every contributor had to run `cd CK_app/CK_app` right after cloning — both README.md and CONTRIBUTING.md had an explicit warning about it, because people kept getting confused and running `yarn install` at the repo root (where there was no `package.json` at all, just a stray empty `yarn.lock` left over from someone doing that by mistake).

We moved every app file up one level, so the app root and the repo root are now the same folder. There is no more `CK_app/` subfolder — `package.json`, `src/`, `quasar.config.js`, `src-capacitor/`, etc. all live directly at `D:\dev\CK_app\`.

## Why this is worth doing

This wasn't a cosmetic preference — it was a documented, repeated point of confusion (see the git history of README.md/CONTRIBUTING.md, which had to add warning callouts about it). Concretely, flattening:
- Removes the double `cd CK_app/CK_app` step and the "which yarn.lock is the real one" trap (there were two — the real one nested inside, and a dead, empty stub at the repo root from an earlier mistaken `yarn install` run there).
- Simplifies the Android/iOS GitHub Actions workflows: they no longer need a `working-directory: ./CK_app` override or a `CK_app/` prefix on every cache/artifact/trigger path.
- Matches how almost every single-app repo is structured — most tooling (editors, linters, CI providers) assumes `package.json` sits at the repo root unless told otherwise.

## How the move was done (for anyone curious about the mechanics)

Every tracked file was moved with `git mv`, not deleted-and-recreated — this preserves each file's history (`git log --follow` / `git blame` still work across the move). The two `.gitignore` files (one at the repo root, one inside the old `CK_app/`) were merged into a single one at the new root, since gitignore patterns are relative to wherever the `.gitignore` file itself lives — anchored patterns like `/src-capacitor/node_modules` needed to end up in a `.gitignore` that's now sitting in the same place `src-capacitor/` itself sits.

Untracked build output (`node_modules/`, `.quasar/`) wasn't moved — those are regenerated by `yarn install` and `quasar dev`/`quasar build` respectively, so they were simply deleted from the old location and rebuilt fresh at the new one instead of moving potentially gigabytes of generated files.

## What else had to change

Beyond the file move itself, anything that *referenced* the old `CK_app/` path had to be updated:
- **`.github/workflows/build_android.yml` and `build_ios.yml`**: removed `working-directory: ./CK_app`, and dropped the `CK_app/` prefix from trigger paths, the Gradle/yarn cache dependency paths, and the build-artifact output paths (7 places total across both files).
- **README.md / README.en.md / CONTRIBUTING.md / CONTRIBUTING.en.md** (both languages): removed the `cd CK_app/CK_app` instructions and the "`../tools/` lives alongside `CK_app/`" framing (now `tools/` and the app's `src/` are just two ordinary sibling folders at the repo root — there's no longer an "outside the project" distinction to explain). Also fixed a couple of already-stale doc lines found along the way (the `src/boot/` file listing still mentioned `firebase.js`, which Phase 3 had already deleted).
- **`quasar.config.js` / `.env.example`**: updated code comments that mentioned the old `CK_app/.env.local` path.
- **`docs/refactoring-plan.md`**: updated the still-pending "version alignment" task's path (a Phase 4+ item), since it will be acted on after this change. A separate, already-completed Phase 0 line describing a past move was deliberately left alone — it's an accurate historical record of what existed at the time, not a live instruction.

**What was deliberately left unchanged:** every plan and decision document under `docs/superpowers/plans/` and `docs/decisions/phase-3-remove-login/` still refers to the old `CK_app/` paths. These are point-in-time records of work already done — rewriting them to match a structure that didn't exist yet when they were written would make the history less accurate, not more.

## Verification performed

- `yarn install --frozen-lockfile`, `yarn lint`, and `yarn dev` all run clean from the new root.
- Spot-checked 5 routes (`/`, `/#/settings`, `/#/food`, `/#/schedule`, `/#/news`) — all HTTP 200, no console/compile errors.
- Confirmed every remaining textual match for `CK_app/` in the repo is either a GitHub URL (`github.com/CK-APP-Org/CK_app/...` — the repo's actual name, unrelated to folder structure) or a historical plan/decision doc (deliberately left as-is, see above).

## What's not verified

The Android/iOS GitHub Actions build-and-deploy pipelines were reasoned through carefully (their `run:` steps already used paths relative to the working directory, like `src-capacitor/android/gradlew`, so removing the `CK_app/` working-directory override should make them resolve correctly against the new root) but could not be executed locally — there's no macOS/Android CI environment available here. Recommended: trigger a manual `workflow_dispatch` run (without the `[deploy]` commit prefix) to confirm both pipelines still build successfully before trusting them on a real release.
